# RAG 기반 RFQ 생성 컴포넌트 (PoC)

> 영림원 춘계공모전 대상作 — ERP 전표 기반 경량 SRM의 **RAG 기반 RFQ 생성** 컴포넌트를 단독 구현한 PoC

구매요청 정보를 입력하면, 법무가 사전 승인한 표준계약 조항을 Vector DB에서 검색해 **출처가 명시된 RFQ(견적요청서) 초안**을 생성합니다. 핵심은 RAG로 LLM의 Hallucination을 차단해, 계약 문구가 지어내지지 않도록 보장하는 것입니다.

**🔗 배포 데모:** https://rag-rfq-generator-qxom9n28mmxdbdd7m8eefu.streamlit.app/

## 문제 정의 (AS-IS → TO-BE)

| | 기존 방식 (AS-IS) | 제안 방식 (TO-BE) |
|---|---|---|
| 작업 주체 | 구매 담당자가 직접 | 상태 그래프가 자동 |
| 조항 확보 | 표준계약서 파일을 일일이 뒤져 복사 | Vector DB에서 관련 조항 자동 검색 |
| 출처 | 수기로 확인, 누락 위험 | 사용된 표준조항을 자동 명시 |
| 위험 | 잘못된 조항 인용·문구 변형 | 근거 없는 내용은 생성 차단 |

구매 담당자가 표준계약서를 뒤져 RFQ를 수작성하던 흐름을, **요청 정보 입력 → 관련 표준조항 검색 → 출처 달린 초안 즉시 생성**으로 대체합니다.

## 핵심 특징

- **출처 기반 생성** — 생성된 모든 거래조건에 근거가 된 표준조항(`제N조`)을 함께 표시
- **Hallucination 차단** — 검색된 근거에 없는 내용은 LLM이 생성하지 않도록 프롬프트로 통제하고, 검색 결과가 0건이면 생성 자체를 막음
- **입력 반응형 멀티쿼리 검색** — 품목을 거래유형으로 분류하고 거래조건 축별(대금지급·품질보증·하자담보·납기 등)로 검색어를 생성해, 단일 고정 쿼리의 편향을 줄이고 여러 조항을 고르게 검색
- **입력값 무결성 보장** — 사용자가 입력한 품목·수량·납기는 LLM을 거치지 않고 코드가 직접 조립해, 날짜·수량이 변형되는 문제를 원천 차단
- **조문 단위 검색** — 표준계약서를 `제N조(제목)` 단위로 분할하고 메타데이터를 부여해 정확한 출처 추적
- **두 층위의 방어** — 그래프 레벨에서 필수항목 누락을 분기로 걸러내고(라우터), 노드 내부에서 검색 0건·API 실패를 가드로 처리

## RFQ 문서 구조

생성되는 RFQ는 3개 파트로 구성됩니다.

| 파트 | 내용 | 생성 방식 |
|---|---|---|
| ① 구매품목 정보 | 품목·수량·납기·예산 | 사용자 입력값을 코드가 그대로 삽입 |
| ② 거래조건 | 대금지급·품질보증·하자담보·지체상금 등 | RAG 검색 + LLM 생성 (출처 명시) |
| ③ 공급사 기재란 | 단가·견적·납품가능일 등 | 빈칸 (공급사가 작성, LLM 생성 금지) |

이 분리가 설계의 핵심입니다. **틀리면 안 되는 값(①)은 LLM에 보여주지 않고, 생성이 필요한 부분(②)만 RAG로 처리**합니다.

## 아키텍처

LangGraph 기반의 **결정론적 상태 그래프(에이전틱 워크플로우)**로 구현했습니다. `analyze → search → generate` 3개 노드로 구성되며, `analyze` 직후 필수항목 누락 여부에 따라 분기합니다.

```
[구매요청 입력]
  품목 · 수량 · 납기 · 예산
        │
        ▼
  ┌──────────────┐
  │   analyze    │  필수항목(품목·수량·납기) 누락 검사
  └──────────────┘
        │
        ├───── 누락 있음 ─────▶ END (생성 중단)
        │
        │ 통과
        ▼
  ┌──────────────┐
  │   search     │  품목 분류 → 축별 멀티쿼리 생성 → 검색 결과 합쳐 dedup (※)
  └──────────────┘
        │
        ▼
  ┌──────────────┐
  │  generate    │  ┌ 가드① 검색 0건 시 생성 차단 (Hallucination 방어)
  │              │  └ 가드② LLM 호출 실패 시 try/except
  └──────────────┘
        │  · 거래조건②: 검색된 조항 근거 + LLM 생성
        │  · 입력값①·빈칸③: LLM 우회, 코드가 f-string으로 직접 조립
        ▼
   [RFQ 초안] ──▶ END
     ① + ② + ③
```

> **(※) 검색 전략:** 사용자 입력 품목을 거래유형(원자재·설비·용역·기타)으로 분류한 뒤, 거래조건의 여러 축(대금지급·품질보증·하자담보·납기 등)을 각각 겨냥한 검색어를 생성해 멀티쿼리로 질의합니다. 같은 조항이 중복되면 `dedup_by_article`로 제거하고 상위 N개만 사용합니다. 단일 고정 쿼리는 검색어가 한 주제로 쏠리는 편향이 있는데, 축별 멀티쿼리로 이를 완화했습니다. 검색어 생성이 실패하거나 결과가 0건이면 고정 주제어 쿼리로 fallback해, 어떤 입력에도 빈손으로 끝나지 않도록 했습니다. 품목 분류는 LLM 호출 대신 키워드 사전 매칭으로 처리해 API 호출 수를 줄였습니다.

**방어가 두 층위로 나뉜 이유:** `analyze` 다음의 조건부 분기(라우터)는 *그래프 레벨*에서 흐름을 멈추고, `generate` 내부의 가드는 *노드 레벨*에서 잘못된 생성을 막습니다. 라우터가 막는 것(입력 누락)과 가드가 막는 것(검색 실패·API 오류)은 다른 종류의 실패라, 둘 다 필요합니다.

## 실행 방법

### A. 로컬 실행 (Streamlit)

```bash
# 1. 가상환경 활성화 (Windows PowerShell)
python -m venv venv
.\venv\Scripts\Activate.ps1

# 2. 의존성 설치
pip install -r requirements.txt

# 3. API 키 설정 (.env 파일)
# GOOGLE_API_KEY=your_key_here

# 4. (최초 1회) 벡터 DB 구축 — 저장소에 chroma_db/가 포함되어 있어 생략 가능
python rag_langchain.py

# 5. 웹앱 실행
streamlit run app.py
```

### B. Docker로 실행

```bash
# 이미지 빌드
docker build -t rfq-app .

# 컨테이너 실행 (API 키는 런타임에 주입)
docker run -p 8501:8501 -e GOOGLE_API_KEY="your_key_here" rfq-app
# → http://localhost:8501
```

> `.env` 파일 또는 `-e` 옵션으로 `GOOGLE_API_KEY`가 필요합니다. (Gemini API 무료 티어 사용)
> API 키는 이미지에 굽지 않고 런타임에 주입합니다.

## 한계 및 확장 방향

이 PoC는 15일 단독 구현으로, 데모에 필요한 핵심 파이프라인을 검증하는 데 초점을 맞췄습니다. 현 시점의 한계와 확장 방향을 솔직히 정리합니다.

| | 현재 | 확장 방향 |
|---|---|---|
| 검색 쿼리 | 품목 분류 기반 입력 반응형 멀티쿼리 (구현 완료) | 검색어 편향 보정·축별 가중치 튜닝, 쿼리 생성 품질 평가 |
| 코퍼스 | 단일 범용 표준계약서 (174벡터) | 거래유형별 코퍼스 분리 + 유형에 따른 검색 라우팅 |
| 조항 출처 | 공개 표준양식(공정위·법제처) | 사내 법무가 승인한 표준조항 DB |
| 워크플로우 | 3노드 상태 그래프 (선형 + 단일 분기) | 조건 분기·재시도·도구 호출을 갖춘 다단계 그래프 |

> **선제적 인정:** 코퍼스가 범용 표준계약서 1종이라, 검색 결과의 다양성은 제한적입니다. 이 PoC의 의의는 *지금의 출력 다양성*이 아니라, 코퍼스가 사내 수백 개 조항으로 확장됐을 때를 전제한 **파이프라인 구조(입력 반응형 멀티쿼리 검색 → 근거 기반 생성 → 출처 추적 → 무결성 분리)가 동작함을 검증**한 데 있습니다. 검색어 생성·검색·거래조건 정리에 각각 모델을 사용하므로, 코퍼스가 커질수록 "어느 조항이 이 거래에 맞는가"를 판단하고 정리하는 모델의 역할이 본격적으로 커집니다.
>
> **검색 편향에 대하여:** 축별 멀티쿼리로 단일 쿼리의 쏠림을 완화했으나, 생성된 검색어가 특정 주제(예: 납기·지연)로 치우치면 관련 조항이 과다 검색될 수 있습니다. 근본적으로는 거래유형별 코퍼스 분리로 해소할 영역이며, 위 확장 방향에 반영했습니다.
